# Amplitude에서 볼 수 있는 이벤트 — 읽는 사람용 정리

기준 2026-07-20 · 워크스페이스 `dawn-cake-773363` · https://hanam-bbq-tayho.vercel.app

Meta 광고 랜딩페이지에 심어둔 커스텀 이벤트 8종이 **실제로 어떤 행동을 뜻하는지**,
그리고 광고 성과를 판단할 때 무엇을 보면 되는지 정리했다.

속성 값의 전체 목록·구현 메모는 [amplitude-taxonomy.md](amplitude-taxonomy.md) 참조.
이 문서는 그 스펙을 분석 담당자·광고주가 읽을 수 있게 요약한 것이다.

---

## 1. 전환 이벤트 — 광고비를 판단하는 기준

모두 `CTA Clicked` 하나로 들어오고, `cta_type` 속성으로 어떤 행동인지 구분한다.

| cta_type | 실제 행동 | 비중 | 버튼 위치 |
|---|---|---|---|
| **`message_book`** | 메신저로 예약 문의 | **주 전환** | 상단바 · 첫 화면 · 하단 |
| **`call_phone`** | 전화 걸기 | **주 전환** | 하단 · 우하단 고정 버튼 |
| `directions_reserve` | 구글맵 열기 (위치 확인 = 방문 의향) | 중간 지표 | 첫 화면 · 하단 · 지도 탭 |
| `visit_facebook` | 페이스북 페이지 방문 | 관심 신호 | 하단 텍스트 링크 |
| `view_menu` | 메뉴 섹션으로 스크롤 (전환 아님) | 탐색 | 첫 화면 |

## 2. 관심도 이벤트 — 무엇에 반응했는가

예약까지 가지 않아도, 메뉴 구성·가격 전략의 근거가 된다.

| 이벤트 | 실제 행동 | 핵심 속성 | 이렇게 쓴다 |
|---|---|---|---|
| **`Menu Item Viewed`** | 메뉴 카드를 눌러 상세 팝업 열기 | `item_name_ko`<br>`item_price_vnd`<br>`item_category` | 인기 메뉴 순위, 어느 가격대에 반응하는지 |
| `Menu Tab Switched` | 탭 전환 (인기·메인·사이드·주류) | `menu_tab` | 여러 탭을 볼수록 진지한 방문 의향 |
| `Menu Group Jumped` | 그룹 점프 칩 (콤보·돼지·소·런치) | `menu_group` | 어떤 육류를 찾아 들어왔는지 |

## 3. 보조 이벤트

단독으로 보기보다, 위 지표를 해석할 때 곁들이는 정보.

| 이벤트 | 실제 행동 | 쓸모 |
|---|---|---|
| `Language Switched` | VI ↔ EN 전환 | 베트남 현지인 대 외국인 비중 |
| `Navigation Clicked` | 상단 메뉴 클릭 | 어느 섹션에 관심이 쏠리는지 |
| `Menu Modal Closed` | 상세 팝업 닫기 | 닫는 방식으로 UX 점검 |
| `Privacy Policy Viewed` | 개인정보 정책 열람 | 거의 발생하지 않음 |

## 4. 모든 이벤트에 함께 붙는 속성

이 값들로 쪼개 보는 것이 분석의 대부분이다.

| 속성 | 값 | 이렇게 쓴다 |
|---|---|---|
| **`utm_term`** | 광고세트명 | 타겟별 성과 비교 — 어느 타겟이 문의로 이어지는지 |
| **`utm_content`** | 소재명 | 소재별 성과 비교 |
| `utm_campaign` | 캠페인명 | 캠페인 단위 집계 |
| `initial_utm_*` | 최초 유입값 | 첫 유입 경로 (한 번 기록되면 고정) |
| `language` | `vi` / `en` | 언어별 행동 차이 |
| `device_type` | `mobile` / `desktop` | 광고 유입은 대부분 모바일 |
| `Country` | 국가 (자동) | **봇·QA 트래픽 제외에 필수** |

> Amplitude 가 잡는 UTM 은 **표준 5개뿐**(source/medium/campaign/term/content).
> `utm_adset` 같은 임의 키는 유저 속성이 되지 않으므로 광고세트는 `utm_term` 에 넣는다.

## 5. 분석 시작하기

1. **세그먼트부터 건다** — `Country = Vietnam`
   현재 쌓인 데이터는 광고 심사·크롤러·QA 접속이라 실제 고객이 없다.
   안 걸면 실제 고객 수십 명이 봇 수천 건에 묻힌다.

2. **퍼널을 만든다**
   ```
   Page Viewed  →  Menu Item Viewed  →  CTA Clicked
                                        └ 필터  cta_type = message_book
   ```

3. **광고세트로 쪼갠다** — 퍼널에 `utm_term` 으로 그룹화
   어느 타겟이 실제 문의까지 오는지 비교된다.

## 6. 알고 있어야 할 한계

이 네 가지를 모르면 숫자를 잘못 읽는다.

- **버튼 클릭까지만 기록된다.** 메신저로 넘어간 뒤 실제로 메시지를 보냈는지,
  전화가 연결됐는지는 알 수 없다. 받은편지함 실제 문의 수를 주 1회 세어
  비율을 잡아두면 추정이 가능하다.
- **실제 예약·매출·ROAS 는 알 수 없다.** 광고세트 간 *상대 비교*는 되지만
  광고비 대비 남는지는 판단 불가. 매장에서 "웹사이트 보고 오셨나요?"를
  체크하면 절반은 메울 수 있다.
- **스크롤 깊이는 측정되지 않는다.** 어디서 이탈하는지 모른다. 다만 전화·메신저
  버튼이 화면에 항상 고정돼 있어 전환 경로가 스크롤에 막히지는 않는다.
  필요해지면 그때 추가하면 된다.
- **세션 리플레이는 10% 만 기록된다.** 특정 방문을 리플레이에서 찾지 못할 수 있다.
  이벤트 자체는 100% 수집된다.
